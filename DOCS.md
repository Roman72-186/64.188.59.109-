## 1. Что это и зачем

У шалам.io нет входящего webhook, поэтому нельзя напрямую «дёрнуть сценарий» после оплаты.
Прокладка решает это так: принимает оплату (эквайринг Т-Банка для карты/СБП/рассрочки и
прямой Partner API Долями для Долями), а после успешной оплаты назначает контакту тег
через API шалам.io. Внутри платформы авторассылка стартует по этому тегу.

**Разделение ответственности:**
- прокладка отвечает за оплату и безопасность;
- шалам.io отвечает за запуск сценариев по тегам.

Стек: Python + FastAPI + SQLite. Касса/чеки не используются.

## 2. Как работает (схема)

```
Пользователь нажимает кнопку оплаты в боте
        │
        ▼
Бот шалам.io → POST /init-payment на прокладку
        │
        ▼
Прокладка сохраняет заказ в БД и создаёт платёж в Т-Банке
        │
        ▼
Т-Банк возвращает ссылку → прокладка отдаёт её боту
        │
        ▼
Бот отправляет ссылку пользователю → пользователь платит
        │
        ▼
Т-Банк → webhook POST /webhook/tbank на прокладку
        │
        ▼
Прокладка проверяет подпись, статус, сумму, дубли
        │
        ▼
Прокладка через API шалам.io ставит контакту тег (+ переменные)
        │
        ▼
Авторассылка в шалам.io стартует по тегу
        │
        ▼
Первый шаг авторассылки сам удаляет тег (НЕ прокладка!)
```

**Важно по тегам:** прокладка тег только назначает. Удаление тега делается
**внутри авторассылки** на первом шаге — так меньше риск, что тег снимется
раньше, чем платформа успеет запустить сценарий.

## 5. Настройка config.yaml

Это единственный файл, который правится под конкретный проект. Код трогать не нужно.

### Блок доступов

```yaml
tbank:
  terminal_key: "..."        # TerminalKey из ЛК Т-Банка
  terminal_password: "..."   # пароль терминала
  api_url: "https://securepay.tinkoff.ru/v2"   # боевой
  # тестовый: "https://rest-api-test.tinkoff.ru/v2"

shalamo:
  api_url: "https://api.shalamov.io"   # базовый URL API шалам.io
  api_key: "..."                       # ключ API шалам.io

server:
  host: "0.0.0.0"
  port: 8000
  public_url: "https://your-domain.ru"   # публичный адрес прокладки
  secret_token: "..."   # см. ниже
```

**Сгенерировать `secret_token`:**

```bash
python3 -c "import secrets; print(secrets.token_hex(32))"
```

Этот токен защищает `/init-payment`. Его же надо вписать в HTTP-запрос бота
в шалам.io как заголовок `X-Secret-Token`.

### Блок товаров

Ключ словаря — это `product_id`, который бот передаёт в запросе.

```yaml
products:
  course_basic:
    name: "Курс: Базовый тариф"
    amount: 9900            # СУММА В КОПЕЙКАХ (9900 = 99 руб)
    description: "Базовый тариф курса"
    payment_methods:        # какие способы оплаты разрешены этому товару
      - card
      - sbp
      - dolyami
    tags_by_method:         # какой тег ставить за какой способ оплаты
      card: "paid_card_basic"
      sbp: "paid_sbp_basic"
      dolyami: "paid_dolyami_basic"
    variables:              # доп. переменные контакта (опционально)
      product_name: "Базовый тариф"
      access_level: "basic"
```

> ⚠️ `amount` всегда в копейках. 99 рублей = `9900`.

Новый товар добавляется копированием блока. Кода это не касается.

### Блок способов оплаты

```yaml
payment_methods:
  card:
    label: "Банковская карта"
    extra_params: {}
  sbp:
    label: "СБП"
    extra_params:
      PayType: "O"
  dolyami:
    label: "Долями"
    extra_params:
      PayType: "O"
  installment:
    label: "Рассрочка"
    extra_params:
      PayType: "O"
```

### Блок доп. терминалов (один способ оплаты на форме)

> Это про способы **эквайринга Т-Банка** (карта/СБП/рассрочка). Для **Долями** изоляция
> формы решена иначе — отдельным провайдером (прямой Partner API, см. блок выше); через
> доп. терминал Т-Банка её вести больше не нужно.

Какие способы оплаты показывает платёжная форма Т-Банка, определяет **сам терминал**,
а не запрос `Init` — в API нет параметра «показать только один способ». Чтобы форма
показывала **только один способ** (напр. отдельная кнопка «Рассрочка»), заведи
в Т-Банке **отдельный магазин/терминал, где включён лишь этот способ**, и направь на
него нужный `payment_method`:

```yaml
tbank:
  terminal_key: "..."          # основной терминал (карта/СБП/…)
  terminal_password: "..."
  api_url: "https://securepay.tinkoff.ru/v2"
  extra_terminals:             # доп. терминалы (отдельные магазины)
    installment_shop:
      terminal_key: "..."      # терминал, где включён ТОЛЬКО нужный способ
      terminal_password: "..." # api_url/timeout наследуются от основного, если не заданы

payment_methods:
  installment:
    label: "Рассрочка"
    extra_params: { PayType: "O" }
    terminal: installment_shop # ← вести этот способ через доп. терминал
```

- Способы без `terminal` идут через основной терминал — поведение без `extra_terminals`
  не меняется (обратная совместимость).
- `Init`/`GetState` для способа уходят на его терминал; **подпись webhook проверяется
  по `TerminalKey` из тела** — каждый магазин подписан своим паролем (поэтому одного
  пароля недостаточно). Webhook-URL общий: `public_url` + `/webhook/tbank`.
- Касса (54-ФЗ): одну онлайн-кассу можно подключить к нескольким магазинам, поэтому
  `Receipt` остаётся общим. Убедись, что к доп. магазину подключена касса, иначе его
  Init упрётся в ту же ошибку `309 {request.validate.expected.receipt}`.

### Блок чека (Receipt, 54-ФЗ)

Боевой терминал Т-Банка с подключённой онлайн-кассой **требует объект `Receipt`**
в каждом Init. Без него Init отклоняется с ошибкой
`309 {request.validate.expected.receipt}`. Терминал без фискализации (и тестовый
контур) чек не требует — тогда блок `receipt` не нужен (или `enabled: false`).

```yaml
receipt:
  enabled: true
  taxation: "usn_income"      # система налогообложения — указать СВОЮ
  tax: "none"                 # ставка НДС позиции (для УСН обычно none)
  payment_object: "service"   # признак предмета расчёта (курс/услуга = service)
  payment_method: "full_prepayment"  # признак способа расчёта (предоплата)
  email: ""                   # fallback-email для чека, если бот не передал
  phone: ""                   # fallback-телефон
```

- Чек собирается автоматически из **одной позиции** = оплачиваемый товар:
  `Name`/`Price`/`Amount` берутся из блока `products` (сумма в копейках).
- Ставку НДС можно переопределить на товар: поле `tax` в блоке товара.
- **Контакт получателя чека (Email или Phone) обязателен по 54-ФЗ.** Его шлёт бот
  в `/init-payment` (необязательные поля `email`/`phone`); если бот не передал —
  берётся `receipt.email`/`receipt.phone`. Если ни там, ни там — Т-Банк отклонит Init.
- `Receipt` — вложенный объект и в подпись `Token` **не входит** (так и требует Т-Банк).

### Блок Долями — прямой Partner API (`provider: dolyame`)

Способ «Долями» можно вести **не через эквайринг Т-Банка, а через прямой Partner API
Долями** (`partner.dolyame.ru`). Зачем: прямой API даёт нативную **Долями-only форму**
(в эквайринге обязательные способы из формы не убираются). Это отдельный провайдер,
включается на уровне способа оплаты полем `provider: dolyame`:

```yaml
payment_methods:
  dolyami:
    label: "Долями"
    provider: dolyame          # ← вести через прямой Partner API (по умолчанию tbank)
    extra_params: { PayType: "O" }   # extra_params для эквайринга; провайдером dolyame игнорируются

dolyame:
  base_url: "https://partner.dolyame.ru"
  login: "<boevoy_login>"      # HTTP Basic login (из ЛК Т-Бизнеса)
  password: "<boevoy_password>"
  cert_path: "/opt/tbank_proxy/certs/<cert>.pem"  # mTLS-сертификат, АБСОЛЮТНЫЙ путь
  key_path:  "/opt/tbank_proxy/certs/<key>.key"   # приватный ключ к нему
  timeout_seconds: 5           # на 429 уважается заголовок X-Retry-After
  fiscalization: disabled      # disabled | enabled (объект fiscalization_settings)
  commit_on_webhook: true      # true: на webhook делать commit (захват денег) → тег
  webhook_allowed_subnet: "91.194.226.0/23"   # подсеть, с которой принимаем webhook Долями
```

- **mTLS обязателен на КАЖДЫЙ запрос.** Без клиентского сертификата не пройдёт даже
  TLS-handshake. Сертификат + ключ держать в **постоянном** месте (не `/tmp`),
  права `600`/`640`, владелец — **пользователь сервиса** (`www-data`); каталог должен
  быть доступен ему на чтение/проход. `cert_path`/`key_path` — **абсолютные** (относительные
  резолвятся от `WorkingDirectory` сервиса и легко ломаются).
- **`provider: dolyame` требует блок `dolyame`** в конфиге (иначе приложение упадёт на
  старте с понятной ошибкой). Способы без `provider` идут через эквайринг (`tbank`).
- **Суммы.** В API Долями суммы — **в рублях**; в проекте хранятся в копейках,
  конвертация точная (Decimal). `amount` товара по-прежнему задаётся в копейках.
- **Двухфазность.** Клиент платит первые 25% → статус `wait_for_commit` (деньги
  захолдированы) → прокладка вызывает **`commit`** (реальный захват всей суммы) → `committed`
  → ставится тег. `commit_on_webhook: false` — ставить тег уже на холде, без захвата.
- **Фискализация.** `disabled` — чек не шлём (проверено: боевой checkout Долями проходит
  без чека). Если конкретный мерчант потребует чек — `enabled` + доработка адаптера
  (полноценный объект `fiscalization_settings`).
- **Креды и сертификат — секреты.** Если фигурировали в переписке — ротировать у менеджера.

> ⚠️ `commit` на webhook = **реальное списание** у клиента. Возвраты вне scope прокладки
> (метод `refund` в адаптере есть, но не используется). Первый боевой прогон — под наблюдением.

> 💡 **Минимум суммы и скоринг.** API-`create` принимает небольшие суммы, но финальное
> одобрение делает **банк/скоринг Долями** на стороне покупателя (может прийти `rejected`).
> Телефон покупателя в форме нужен в формате `+79991234567`; неверный телефон даёт на
> форме generic-ошибку «сервис временно недоступен».

## 6. Запуск и автозапуск

### Ручной запуск (для проверки)

```bash
source venv/bin/activate
uvicorn app.main:create_app --factory --host 0.0.0.0 --port 8000
```

Проверка:

```bash
curl http://localhost:8000/health
# {"status":"ok"}
```

### Автозапуск через systemd

Создай `/etc/systemd/system/tbank-proxy.service`:

```ini
[Unit]
Description=TBank Shalamo Proxy
After=network.target

[Service]
User=www-data
WorkingDirectory=/opt/tbank_proxy
ExecStart=/opt/tbank_proxy/venv/bin/uvicorn app.main:create_app --factory --host 0.0.0.0 --port 8000
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

```bash
systemctl daemon-reload
systemctl enable tbank-proxy
systemctl start tbank-proxy
systemctl status tbank-proxy
```

## 7. Reverse proxy + HTTPS

Т-Банк шлёт webhook только на HTTPS. Нужен nginx (или caddy) перед прокладкой.

Пример nginx:

```nginx
server {
    listen 443 ssl;
    server_name your-domain.ru;

    ssl_certificate     /etc/letsencrypt/live/your-domain.ru/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/your-domain.ru/privkey.pem;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

Сертификат — через `certbot --nginx`.

> ⚠️ Заголовок `X-Real-IP $remote_addr` (строка выше) **обязателен** для `/webhook/dolyame`:
> по нему проверяется, что webhook пришёл из подсети Долями (`webhook_allowed_subnet`).
> Без него реальный IP отправителя теряется и webhook будет отвергаться с `403`.

## 8. Настройка Т-Банка

1. В ЛК Т-Банка возьми `TerminalKey` и пароль терминала → впиши в `config.yaml`.
2. Укажи URL уведомлений (webhook):
   ```
   https://your-domain.ru/webhook/tbank
   ```
3. Убедись, что нужные способы **эквайринга** подключены к терминалу:
   карта, СБП, Рассрочка. (Долями — отдельный провайдер, настраивается не здесь, а
   в блоке `dolyame`, см. §5.)

> ⚠️ **Важно про Рассрочку.** Она работает через платёжную форму Т-Банка (прокладка
> её и использует), но должна быть **отдельно активирована на терминале**. Если не
> активирована — ссылка просто не предложит этот способ. Подтверди подключение у
> менеджера Т-Банка.
>
> **Долями** через эквайринг Т-Банка больше не ведётся — это отдельный провайдер
> (прямой Partner API): настройка mTLS-сертификата, кредов и webhook — см. §5 и §7.

## 9. Настройка шалам.io

### 9.1 Настроить адаптер shalamov.io (в config.yaml, НЕ в коде)

`app/shalamo.py` — конфиг-адаптер: путь, метод, тело запроса и схема авторизации
берутся из блока `shalamo` в `config.yaml`. **Код трогать не нужно** — когда узнаешь
реальный контракт API shalamov.io, правишь только конфиг:

```yaml
shalamo:
  api_url: "https://api.shalamov.io"
  api_key: "..."
  timeout_seconds: 3            # *2 попытки должно укладываться в таймаут webhook (~10с)
  auth:
    header: "Authorization"
    value_template: "Bearer {api_key}"   # поправь, если авторизация иная
  assign_tag:
    method: "POST"
    path: "/contacts/tag"                # ← реальный endpoint shalamov.io
    body_template: { contact_id: "{contact_id}", tag: "{tag}" }
  set_variables:
    method: "POST"
    path: "/contacts/variables"          # ← реальный endpoint shalamov.io
    body_template: { contact_id: "{contact_id}", variables: "{variables}" }
```

Плейсхолдеры: `{contact_id}`, `{tag}` — строки; `{variables}` подставляется как
объект (dict). Уточни у shalamov.io точный путь, формат тела и способ авторизации.

> Тег — «гейт» доступа: именно его успешная установка запускает авторассылку.
> Переменные отправляются перед тегом (best-effort). Если переменные не
> установились, но тег встал — доступ считается выданным, факт пишется в лог.

### 9.2 Настроить бота

Бот при нажатии кнопки оплаты шлёт POST на прокладку:

```
POST https://your-domain.ru/init-payment
Content-Type: application/json
X-Secret-Token: <тот же токен, что в config.yaml>

{
  "contact_id": "{{contact_id}}",
  "product_id": "course_basic",
  "payment_method": "card"
}
```

- `contact_id` — **внутренний ID контакта шалам.io** (не телефон, не username).
  Подставляется переменной бота. Без него тег некому ставить.
- В ответ бот получает `pay_url` → отправляет ссылку пользователю.

### 9.3 Настроить авторассылки

Под каждый тег из `config.yaml` должна быть авторассылка в шалам.io.
**Первый шаг каждой авторассылки — удаление своего тега** (чтобы пользователь
повторно не попал в цепочку).

## 10. API прокладки (справочник)

### `GET /health`

Проверка живости. Возвращает `{"status":"ok"}`.

### `POST /init-payment`

Создаёт платёж, возвращает ссылку на оплату.

**Заголовки:**

| Заголовок | Значение |
|-----------|----------|
| `Content-Type` | `application/json` |
| `X-Secret-Token` | секретный токен из config.yaml |

**Тело:**

| Поле | Тип | Описание |
|------|-----|----------|
| `contact_id` | строка | Внутренний ID контакта шалам.io |
| `product_id` | строка | ID товара из config.yaml |
| `payment_method` | строка | `card` / `sbp` / `dolyami` / `installment` |

**Ответ — единый формат со статусом** (`status`, плюс `order_id`/`pay_url`, где есть ссылка):

```json
{
  "status": "created",
  "order_id": "course_basic_12345_a3f8c1d2",
  "pay_url": "https://securepay.tinkoff.ru/..."
}
```

| `status` | HTTP | Значение |
|----------|------|----------|
| `created` | 200 | создан новый платёж и ссылка |
| `existing_active` | 200 | возвращена активная неоплаченная ссылка (младше 1 ч) |
| `already_paid_access_granted` | 200 | товар уже оплачен, тег доступа назначен (или назначен сейчас) |
| `already_paid_pending_access` | 200 | товар оплачен, но тег пока не удалось назначить |
| `invalid_product` | 400 | неизвестный `product_id` |
| `invalid_payment_method` | 400 | способ недоступен товару |
| `payment_creation_failed` | 502 | провайдер не создал платёж (Т-Банк Init или Долями create) |
| `forbidden` | 403 | неверен/отсутствует `X-Secret-Token` |

### `POST /webhook/tbank`

Принимает уведомления Т-Банка. Вызывается **самим Т-Банком**, не ботом.
Защита — подпись Token от Т-Банка.

Внутри по порядку: подпись → наличие заказа → сохранение статуса → (если
`CONFIRMED`) сверка суммы (при расхождении — доппроверка `GetState`) →
идемпотентность (тег уже назначен?) → фиксация оплаты → атомарный захват →
**синхронное назначение тега, до 2 быстрых попыток**.

- успех → HTTP `200` тело `OK`;
- статус не `CONFIRMED` или плохая подпись/сумма → `200` `OK`, доступ не выдаётся;
- shalamo недоступен (обе попытки неуспешны) → **HTTP `503` без `OK`**, платёж
  остаётся пере-обрабатываемым, Т-Банк повторит webhook (PRD §7.8). Тег при этом
  НЕ назначается дважды — гарантирует `tag_assigned_at`.

### `POST /webhook/dolyame`

Принимает уведомления **прямого Partner API Долями** (когда способ идёт через
`provider: dolyame`). Вызывается серверами Долями, не ботом.

- **Защита — IP-allowlist**, а не подпись: тело webhook Долями не подписано (как и у
  Т-Банка), поэтому источником истины служит **`GET /info`**, а доверенность отправителя —
  по IP из подсети `webhook_allowed_subnet` (по умолчанию `91.194.226.0/23`). Реальный IP
  берётся из `X-Real-IP` (ставит nginx) — **обязательно проксировать этот заголовок**
  (см. §7). Чужой IP → `403`.
- Внутри по порядку: проверка IP → поиск заказа по `id` → **`GET /info`** (статус/сумма) →
  негативный статус (`rejected`/`canceled`) → `200`, доступ не выдаётся → сверка суммы
  (рубли, Decimal) → на `wait_for_commit` синхронный **`commit`** (если `commit_on_webhook`)
  → идемпотентность → атомарный захват → назначение тега (1 попытка).
- успех → HTTP `200` `OK`;
- `/info` недоступен / `commit` не удался / тег не назначен → **HTTP `503`** — Долями
  повторит webhook (`commit` идемпотентен: на повторе статус уже `committed`, повторно не зовётся).

### Переменные, которые получает контакт после оплаты

| Переменная | Пример | Описание |
|------------|--------|----------|
| `payment_status` | `paid` | Статус |
| `payment_method` | `card` | Способ оплаты |
| `payment_amount` | `99` | Сумма в рублях |
| `payment_order_id` | `course_basic_12345_a3f8c1d2` | ID заказа |
| `payment_id` | `987654321` | ID платежа в Т-Банке |
| + переменные из `variables` товара | | напр. `access_level` |

**Ключевые строки лога:**

| Строка | Что значит |
|--------|------------|

| `✅ Платёж подтверждён` | тег назначен, всё ок |

`pending` → `processing` → `confirmed` (успех) или `failed`.

Посмотреть платежи:

```bash
sqlite3 payments.db "SELECT order_id, status, tag_name, created_at FROM payments ORDER BY id DESC LIMIT 20;"

## 13. Траблшутинг

**`/init-payment` отдаёт 403.**
Не совпадает `X-Secret-Token`. Сверь заголовок в шалам.io и `secret_token` в config.yaml.

**`/init-payment` отдаёт 502.**
Т-Банк отклонил Init. Смотри `logs/app.log` — там ответ Т-Банка. Часто: неверный
`TerminalKey`/пароль, или сумма/параметры.

**Платёж прошёл, но тег не назначился (`ТРЕБУЕТ РУЧНОГО РАЗБОРА`).**
Оплата успешна, но shalamov.io не ответил после 2 быстрых попыток. Прокладка вернула
Т-Банку `503`, и Т-Банк будет **повторять webhook по своему расписанию** — при
восстановлении shalamov.io тег назначится автоматически (платёж в статусе
`processing`, `paid_at` проставлен, `tag_assigned_at` пуст). Ручной разбор нужен,
только если Т-Банк исчерпал повторы: назначь тег вручную через API shalamov.io.
Причина обычно в блоке `shalamo` в `config.yaml` (неверный путь/формат — см. §9.1)
или временная недоступность shalamov.io. Найти зависшие платежи:

```bash
sqlite3 payments.db "SELECT order_id, last_error FROM payments WHERE paid_at IS NOT NULL AND tag_assigned_at IS NULL;"
```

**Т-Банк шлёт webhook повторно.**
Прокладка обязана отвечать ровно `OK` с кодом 200. Проверь, что nginx не
подменяет ответ и не режет по таймауту. Дубли при этом безопасны —
идемпотентность не даст назначить тег дважды.

**`database is locked`.**
Включён WAL и таймаут 10 сек — при штатной нагрузке не возникает. Если всё же
видишь — значит очень высокая параллельность; тогда есть смысл перейти на PostgreSQL.

**Авторассылка не стартует, хотя тег назначен.**
Это уже сторона шалам.io. Проверь: тег в авторассылке точно совпадает (регистр,
подчёркивания), авторассылка активна, аудитория настроена так, что реагирует
на уже существующий контакт.
